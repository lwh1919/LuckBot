# LuckBot Spec Workflow

## 文件定位

`spec-workflow.md` 描述一个任务如何在 LuckBot 中从输入推进到收尾。

本文件描述任务载体分流、阶段顺序、每阶段允许动作与禁止动作；全局硬规则见 `standards.md`。

## 任务载体分流

### `micro-spec`

适用场景：

- 小任务
- 单点 bug
- 局部改动
- 目标明确、风险可控的任务

产物位置：

- `mydocs/micro_specs/`

### `spec`

适用场景：

- 长链路任务
- 跨模块任务
- 需要更完整研究、计划、验证记录的任务
- 需要更强恢复锚点的任务
- 涉及运行时语义、插件机制、会话、记忆、agent loop 的任务

产物位置：

- `mydocs/specs/`

## 阶段流

### 1. 任务理解

输入：

- 用户任务
- `project/README.md`
- 当前任务相关模块真相文档

产出：

- 任务理解
- 当前目标

允许动作：

- 读取 `project/` 与 `workflow/`
- 提炼任务边界
- 判断任务载体

禁止动作：

- 改代码
- 宣布进入 Execute

### 2. 代码导航

输入：

- 相关 `project/*.md`
- 当前代码目录与入口文件

产出：

- 代码入口
- 调用链
- 依赖模块
- 建议阅读顺序

允许动作：

- 搜索代码
- 确认入口、调用链、依赖模块

禁止动作：

- 改代码
- 先写实现再补 plan

### 3. 建立 Spec

创建或更新 `micro-spec` 或 `spec`，至少写清：

- `goal`
- `scope`
- `constraints`
- `approval status`
- `done contract`
- `next actions`

产出：

- 当前任务 spec

允许动作：

- 新建或更新任务文档
- 明确当前任务仍处于 `Pending Approval`
- 发现已有活跃 spec 时直接更新，不重复新建

禁止动作：

- 跳过 spec 直接进实现

### 4. Research

锁定当前事实：

- 当前实现位置
- 现有行为
- 风险点与未知点

产出：

- 事实清单
- 回写后的 spec

允许动作：

- 继续补代码事实
- 标记待核对项

禁止动作：

- 输出实现代码
- 用猜测替代事实

### 5. Plan

形成实现前计划，至少写清：

- 主要改动思路
- 涉及文件
- 执行顺序
- 验证方式

产出：

- 待批准计划

允许动作：

- 输出文件级或模块级改动计划
- 明确验证方式

禁止动作：

- 未批准先改代码

### 6. Approval

输入：

- 已确认计划

产出：

- 精确批准词

允许动作：

- 等待用户回复 `Plan Approved` 或 `批准执行`

禁止动作：

- 将自然语言同意自动解释为批准

### 7. Execute

前置条件：

- 当前任务已有 spec
- 当前任务已有 plan
- 当前轮次已收到 `Plan Approved` 或 `批准执行`

产出：

- 代码改动
- 执行记录
- 回写后的 spec
- 真相文档影响清单

允许动作：

- 按已批准计划施工
- 在 spec 的 `Change Log` 中回写结果
- 每次修改后审查受影响的真相文档，并记录哪些需要更正

禁止动作：

- 脱离已批准范围扩张实现
- 在执行期重新发明需求

### 8. Validation

基于测试、日志、手动验证或人工反馈判断目标是否完成。

产出：

- 验证结论
- 偏差与剩余差距
- 真相文档审查结论

允许动作：

- 跑验证
- 对照 `Done Contract` 判断是否完成
- 核对本轮改动涉及哪些 `project/` 或 `workflow/` 真相文档
- 明确记录哪些无需更正，哪些已经更正，哪些仍待更正

禁止动作：

- 没有验证就宣布完成
- 没有真相文档审查结论就结束任务

### 9. Reverse Sync

发现偏差时，先更新 spec，再继续代码修正或进入下一轮。

产出：

- 修正后的 spec
- 下一轮目标
- 需要回灌到 `project/`、memory 的长期信息
- 需要补写或更正的真相文档清单

允许动作：

- 先回写 spec 中的偏差、剩余差距、下一轮目标
- 先回写真相文档影响，再决定是否补写 `project/` 或 `workflow/`
- 再决定是否进入下一轮 Research / Plan / Execute

禁止动作：

- 发现偏差后只改代码、不更新 spec

## 小任务最小流

1. 读取项目真相
2. 建立 `micro-spec`
3. 给出最小计划
4. 收到 `Plan Approved` 或 `批准执行`
5. 执行
6. 验证并回写
7. 审查并处理真相文档影响

## 大任务扩展流

1. 读取项目真相
2. 建立 `spec`
3. 补充 research 事实
4. 形成更完整 plan
5. 收到 `Plan Approved` 或 `批准执行`
6. 执行
7. 验证并回写
8. 审查并处理真相文档影响
