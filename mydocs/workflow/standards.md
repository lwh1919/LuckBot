# LuckBot Standards

## 文件定位

`standards.md` 定义 LuckBot 任务推进协议的全局硬规则。

## 真相分层规则

- `project/` 负责项目知识真相
- `workflow/` 负责流程协议真相
- `micro_specs/` 与 `specs/` 负责任务真相

## 文档优先规则

- 开始实现前，先读取 `project/`、`workflow/` 与当前任务 spec
- 没有 spec，不进入代码修改
- 文档与代码冲突时，先核对事实，再修正文档或代码
- 新发现的稳定事实优先回写文档，而不是只留在对话中

## 执行门禁规则

- 用户给任务后，先写清任务理解，再进入 spec
- 实现前必须给出明确计划或 checkpoint
- 未得到精确批准词前，不进入代码修改
- 测试、日志或人工反馈暴露偏差时，先重述当前目标与剩余差距，再决定下一轮动作

## 批准词规则

- 只有收到 `Plan Approved` 或 `批准执行`，才允许进入 Execute
- `Implement the plan`、`可以继续`、`搞吧`、`开始改`、`先做掉` 这些表达默认都不等于批准词
- 如果当前轮次只有“分析、解释、设计、审查、给计划”意图，必须停留在非执行阶段

## 违规处理规则

- 如果 agent 在没有批准词时进入了代码修改，视为协议违规
- 发生协议违规后，下一轮必须先回到 spec 与 plan，明确偏差，再继续推进
- 不允许用“已经改了一部分”为理由跳过门禁

## 回写规则

- 新事实回写当前任务 spec
- 稳定模块事实回写 `project/`
- 可复用的长期结论进入长期 memory
- 每次发生代码或流程文档修改后，必须审查受影响的真相文档范围
- 审查结论至少要写清：涉及哪些真相文档、哪些无需更正、哪些需要更正
- 需要更正的真相文档应在当前任务内同步回写；若暂不回写，必须在 spec 中明确记录原因与待办
- `project/` 的写法遵守 `project/_template.md`
- `micro_specs/` 与 `specs/` 的写法遵守各自目录下的 `_template.md`

## 任务载体规则

- 任务载体选择以 `spec-workflow.md` 为准
- 同主题若已有未完成 spec，优先更新原文档，不新建
- 同一主题同一阶段默认只保留一个活跃 spec
- 不论使用哪种载体，完成都必须由证据证明

## 审查规则

- 完成由证据证明，而不是由执行者自行宣布
- 审查对象包括 spec、代码、执行记录、验证结果
- 审查必须识别偏差、隐性破坏与验证缺口
