# Module Truth Template

## 文件定位

`project/_template.md` 是 LuckBot 模块真相文档模板与写法规范。

本模板服务于 `project/` 下的模块真相文档，例如：

- `project/agent-loop.md`
- `project/plugin-system.md`
- `project/skills-system.md`
- `project/mcp-system.md`
- `project/memory-system.md`
- `project/session-system.md`

## 模板语义

模块真相文档遵守以下语义：

- 文档是唯一真相
- 文档是现在进行时
- 文档代表模块逻辑与边界
- 代码是实现层，不替代真相文档
- 文档必须简洁、高信号、低歧义
- 文档必须客观，不写聊天语气、解释语气、自我代入语气
- 文档描述“现在是什么”，不是“作者怎么想”

模块真相文档要回答的是：

- 这个模块现在负责什么
- 这个模块现在怎么工作
- 关键入口在哪里
- 与其它模块的边界是什么

模块真相文档不回答的是：

- 未来想做什么
- 过去是怎么演化来的
- 为什么当时这么设计
- 漫长教学说明

## 写法要求

- 只写当前事实，不写未来设想
- 只写已经存在的行为，不写理想行为
- 使用陈述句，直接写“现在是什么”
- 避免模糊词，例如“可能”“大概”“似乎”“应该”
- 避免错误人称，例如“我们这里”“你可以看到”“我认为”
- 每个关键结论尽量能落到文件、目录、模块、函数、数据路径
- 不确定的地方允许写 `[待核对]`
- 不要把真相文档写成路径清单式导航地图

## 快速生成规则

当 AI 新写一个模块真相文档时，默认按这个顺序处理：

1. 先确认模块边界
2. 再确认真实入口文件
3. 再确认当前真实行为
4. 最后补关键文件、边界、风险、待核对项

默认写法：

- 先写“文件定位”，一句话说明本文记录什么
- 再写“模块职责”，只列当前职责
- 再写“当前真实行为”，按主题分段
- 每一段都尽量落到真实文件、函数、数据流
- 最后补“关键入口”“边界”“风险”
- 若模块耦合，说明即可

默认禁忌：

- 不写“未来会”“后续要”
- 不写“之前是”“后来改成”
- 不写“为了方便”“这样更优雅”
- 不写主观评价
- 不把代码细节堆成逐行解释

## 默认结构

除 `glossary.md` 外，模块真相文档默认包含：

1. `文件定位`
2. `模块职责` 或 `主要内容`
3. `当前真实行为`
4. `关键入口` 或 `关键文件`
5. `边界`

按需增加：

- `当前阶段焦点`
- `当前非目标`
- `风险`
- `待核对项`

## 模板骨架

以下骨架可直接复制到新的模块真相文档中：

```md
# <Module Name> Truth

## 文件定位

`project/<module-name>.md` 记录 LuckBot <模块名> 当前已经实现的真实行为。

本文只写当前代码事实，不写未来产品设想。当前边界是：

- <边界事实 1>
- <边界事实 2>

## 模块职责

LuckBot 当前的 <模块名> 负责：

- <职责 1>
- <职责 2>

## 当前真实行为

### 1. <主题 1>

<直接写当前事实。>

### 2. <主题 2>

<直接写当前事实。>

## 关键入口

- `<path/to/file>`
  - `<symbol or responsibility>`
- `<path/to/file>`
  - `<symbol or responsibility>`

## 边界

- <模块负责什么>
- <模块不负责什么>
- <与其它模块如何分界>

## 风险

- <高风险点 1>
- <高风险点 2>

## 待核对项

- [待核对] <尚未确认但必须显式标记的点>
```

## 快速检查

在提交一个新的模块真相文档前，至少检查：

- 是否全篇都在描述“现在是什么”
- 是否删掉了过去式和未来式叙述
- 是否删掉了主观判断和聊天口吻
- 是否给出了关键文件或关键函数
- 是否明确写出了模块边界
- 是否把不确定内容标成了 `[待核对]`
